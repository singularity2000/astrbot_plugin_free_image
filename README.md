# 图片生成插件 (astrbot_plugin_free_image)

强大的 Astrbot 图片生成插件。不仅支持文生图、图生图，还在API 容灾、自动化配额管理以及大模型工具调用方面达到了工业级标准。

---

## 核心特性全解析

### 1. 工业级 API 容灾管线
插件不再依赖单一的 API 接口，而是构建了一个可配置的动态管线：
- **顺序调度与自动回退**：用户可配置多个节点（如 Vertex AI -> Gemini -> 硅基流动）。系统会按优先级依次尝试，一旦某个节点失效，立即无感切换到下一个。
- **智能退避重试**：
    - **频率限制退避**：当检测到 `429 Resource Exhausted` 错误时，系统会自动启动指数级延迟重试（2s, 4s, 8s... 直至 16s），并伴随随机抖动（Jitter），最大程度规避并发冲突。
    - **普通故障重试**：针对网络波动等偶然错误，提供 3-5 秒的随机延迟重试。
- **Key 自动轮换与负载均衡**：每个 API 节点均支持配置 Key 列表。系统采用 Round-robin 算法在请求间自动轮换 Key，平摊调用额度。

### 2. 深度逆向：Vertex AI 匿名接口
这是本插件最具技术含量的功能之一，允许用户在无需 API Key 的情况下调用 Google 最顶尖的 Gemini 3.1 系列生图模型：
- **浏览器指纹伪装 (Impersonation)**：基于 `curl_cffi` 深度模拟 Chrome、Firefox、Safari 等真实浏览器指纹，成功绕过 Google 的风控检测。
- **会话老化与重建 (Session Aging)**：系统实时监控会话健康度。每完成一定次数（如 5 次）或遇到风控异常时，会自动销毁旧会话并轮换指纹重新建连，保证长期稳定性。
- **Recaptcha 自动化**：后台自动完成 Google 复杂的 Recaptcha 令牌获取与校验流程。

### 3. 全面的多媒体处理能力
- **图生图 (I2I) 进化版**：
    - **多图上下文**：支持同时参考多张图片（最高 5 张），模型将融合多图特征进行创作。
    - **At 即参考**：支持通过 `@用户` 直接提取该用户的头像作为生图底图。
    - **动态图处理**：自动识别 GIF/WebP 动图，并精准提取第一帧作为参考。
- **高清与多格式支持**：
    - **智能分辨率匹配**：支持识别 Prompt 中的 `1K/2K/4K` 关键词并自动调整模型输出清晰度。
    - **视频生图兼容**：针对 OpenAI 兼容端点（如 `OpenAICompatChatProvider`），能够解析并转发 base64 视频流或视频 URL。

### 4. 精准的配额与权限系统 (Persistence)
- **多维度配额**：
    - **永久点数**：由管理员充值或通过签到系统累积，永不过期。
    - **每日固定额度**：每天 0 点自动重置，为普通用户提供基础保障。
- **原子化扣费**：基于 `asyncio.Lock` 确保在高并发环境下次数统计的绝对准确。
- **权限分级**：支持全局管理员、用户黑白名单、群组黑白名单，细粒度控制资源分配。

### 5. LLM Native：大模型协同逻辑
插件不仅仅是一个指令工具，更是 LLM 的一个“感官工具”：
- **语义理解**：LLM 可以根据上下文自主决定是否调用 `image_generation` 工具。
- **提示词自动化扩充**：插件内置扩充指南，能将用户简单的诉求（如“画只猫”）转化为专业的绘图咒语（包含光影、材质、画质描述）。

## 快速上手指南

### 插件安装

将本仓库作为 AstrBot 插件安装/放入插件目录后启用即可完成安装。

### 基础交互
- **文生图**：`#文生图 <您的描述>`（例如：`#文生图 一位在霓虹灯下的赛博朋克少女`）。
- **图生图**：`#图生图 <修改意见>`（可**直接发送图片**或**引用图片**；也支持在消息里 **@某个用户** 来使用 TA 的 QQ 头像作为底图）。
- **模板生图（预设模板）**：发送“**模板名**”（如 `手办化`、`真人化` 等），并同时：
  - 发送/引用一张图片作为底图；或
  - 在同一条消息中 **@某个用户**（将抓取该用户 QQ 头像作为底图）；
  - 若既没有图片也没有 @ 用户，且当前平台能获取到发送者 QQ 头像，则会**自动使用发送者头像**作为底图。
  - 模板名列表可通过 `#画图帮助` 查看，模板内容来自配置项 `prompt_list`。

> 注意：如果你在配置里开启了“指令必须带前缀”（`prefix=true`），则需要使用命令前缀（如 `#`）或 @机器人 唤醒后才能触发。

### 输出与消息形态
- **普通模式**
  - 收到请求：会先回复一条文字提示，例如“收到图生图/文生图请求，正在生成..."
  - 成功：默认保持现有链式回复风格。成功说明中会显示本次成功命中的模型名，例如 `✅ 生成成功 (3.42s) | 个人剩余次数: 12 | 本群剩余次数: 88 | 模型: gemini-3.1-flash-image-preview`。若图片消息设置为“引用回复”，则会尽量保持 `[引用消息][图片...][成功说明]` 的现状；若图片消息设置为“单独发送”，则会先发一条引用回复的成功说明，再单独发送图片消息。
  - 失败：返回失败原因（包含各 Provider 聚合后的错误信息）
- **简洁模式（仅QQ群聊）**
  - 收到请求：尝试给原消息贴 OK 表情，不主动发文字
  - 成功：只发图片（或视频组件），不附带文字说明；日志中仍保留成功说明，并会显示本次成功命中的模型名
  - 失败：仍会发出失败原因

### 成功消息的可配置行为
- `quote_reply_mode`：只影响**图片消息**是否显式引用回复。
  - `始终引用回复`
  - `始终单独发送`
  - `命令引用回复，函数调用单独发送`
  - `命令单独发送，函数调用引用回复`
- `multi_image_send_mode`：只影响**一次成功结果中的多张图片彼此之间**是否分条发送。
  - `始终不分条`：多张图片尽量合并为一条图片消息
  - `始终分条`：多张图片逐张分开发送
  - `群聊不分条，私聊分条`
  - `群聊分条，私聊不分条`
- 规则补充：
  - 单图时，`multi_image_send_mode` 不会改变消息条数。
  - 私聊始终按普通模式处理；即使配置中开启了简洁模式，私聊成功时也会保留成功说明。
  - 普通模式下，成功说明会单独作为引用回复提示用户“图已画好”；若图片消息配置为引用回复，且又是不分条场景，则会保持现有的 `[引用消息][图片...][成功说明]` 合并样式。

### 个人中心
- **签到**：`#画图签到`。
- **查数**：`#画图查询次数`。
- **查预设**：`#画图帮助`（展示当前已加载的所有提示词模板）。

### 函数工具调用

本插件会注册一个 LLM 函数工具：`image_generation`。

- 工具描述 `llm_tool_description` 与 prompt 扩充指南 `llm_prompt_description` 均可在配置中自定义。
- LLM 调用后，插件会启动后台任务并 `stop_event()`，避免 LLM 继续输出导致超时。
- 在 `image_generation()` 中，插件仅通过消息中是否存在 **Image/Reply(Image)** 来决定 `is_i2i`。LLM 工具入口不会把 “@用户头像” 当作图生图输入（因为判定逻辑只检查 Image/Reply）。
  - 有图片组件：图生图
  - 没有图片组件：文生图

这一节对应 `main.py` 的权限判断与 `persistence.py` 的扣费逻辑。

### 1) 管理员（全局管理员）

- 管理员名单来自 AstrBot 全局配置 `admins_id`。
- 管理员 **不受黑白名单/次数/冷却** 等限制，并且“剩余次数”展示为 ∞。

### 2) 黑白名单

- 用户黑名单 `user_blacklist`：优先级最高
- 群黑名单 `group_blacklist`：优先级最高
- 用户白名单 `user_whitelist`：非空时只允许名单内用户使用（管理员除外）
- 群白名单 `group_whitelist`：非空时只允许名单内群使用

### 3) 次数额度的组成与扣费顺序

对应 `PersistenceManager.get_user_count()` / `check_and_deduct_count()`：

- **用户总次数 = 永久次数 + 今日剩余固定额度**
  - 永久次数：管理员增加、签到奖励都会进入这里
  - 每日固定额度：`user_daily_fixed_quota`，每天 0 点刷新

当用户和群额度都开启时（`enable_user_limit=true` 且 `enable_group_limit=true`）：

1. **优先扣群次数**（群还有次数则扣群）
2. 否则扣用户次数
3. 两者都没次数时返回对应提示

### 4) 签到系统的行为边界

对应 `#画图签到`：

- 只有当 `enable_checkin=true` 时签到命令才会生效；否则会提示“本机器人未开启签到功能”。
- 同一天重复签到会提示“今天已经签到过”，并返回当前剩余次数。
- 奖励会**只增加永久次数**（不会直接写入每日额度）。
- 随机签到奖励启用条件：代码里使用 `str(enable_random_checkin).lower() == 'true'` 判断。
  - 这意味着如果配置层把它存成字符串 `"true"` 也能生效；存成布尔 `true` 也能生效。
  - 若你希望更直观，建议在 WebUI 中按布尔开关方式配置。

### 5) 频率限制的影响范围

- 冷却是**全局**的：任何用户触发都会更新 `last_api_call_time`。
- 管理员不走这段检查逻辑，因此管理员不受全局冷却影响。

插件当前内置 Provider 映射见 `providers/__init__.py`，由 `api_pipeline` 的 `__template_key` 选择。

| template_key | 是否需要 API Key | 返回类型 | 备注 |
| :--- | :--- | :--- | :--- |
| `vertex_ai_anonymous` | 否 | 图片 bytes 或错误字符串 | 逆向匿名；依赖 `curl_cffi`；内置 reCAPTCHA 处理、指纹轮换、会话老化。 |
| `gemini` | 是 | 图片 bytes / 图片列表 / 错误字符串 | 支持 `image_size=智能匹配`（从 prompt 里提取 1K/2K/4K）；支持 google_search(仅 gemini-3)。对反代/SSE/噪声响应有兼容解析。 |
| `vertex_ai` | 是 | 图片 bytes / 图片列表 / 错误字符串 | 官方 Vertex AI；请求体更“轻”，不含 image_size/google_search 等扩展字段。 |
| `openai_responses` | 是 | 图片 bytes / 图片列表 / 错误字符串 | 兼容 `/v1/responses` 端点，获取并解析 base64 图片。 |

| `siliconflow` | 是 | 图片 bytes / 图片列表 / 错误字符串 | 通用 `/images/generations`；可解析多个返回图片 URL。可携带最多 3 张输入图。 |
| `bigmodel` | 是 | 图片 bytes / 图片列表 / 错误字符串 | 通用 `/images/generations`；可解析多个返回图片 URL。 |

<details>
<summary>点击展开：提供商详细介绍</summary>

### 1) 重试策略与退避

- 所有 Provider 继承 `BaseProvider`，重试上限来自节点配置 `max_retry`。
- `BaseProvider` 内部统一封装了两类重试延迟：
  - **资源耗尽/429**：2、4、8、16...（最大 16）并叠加 0~3 秒抖动。
  - **普通重试**：3~5 秒随机抖动。

### 2) GeminiProvider（原生 Gemini API）

- 端点形式：`{api_url}/{model}:generateContent`
- 鉴权：`x-goog-api-key`
- 返回解析：
  - 支持标准 JSON 响应
  - 支持 **SSE 风格**的 `data: {...}` 响应（会取最新一条可解析 JSON）
  - 支持“反代在 JSON 前后加噪声”的场景（尝试裁剪首尾 `{}`）
- `image_size=智能匹配`：会从 prompt 里匹配 `1K/2K/4K`，否则默认 `1K`。
- `google_search`：仅在模型名包含 `gemini-3` 时生效（会在请求体加入 `tools: [{google_search:{}}]`）。

### 3) VertexAIProvider（官方 Vertex AI）

- 端点形式：`{api_url}/{model}:generateContent?key={api_key}`
- 返回解析与 Gemini 类似：读取 `candidates[].content.parts[].inlineData.data`。
- 相比 GeminiProvider：请求体更精简（不包含 image_size/google_search/system_prompt 等扩展字段）。

### 4) VertexAIAnonymousProvider（逆向匿名）

- 不需要用户 API Key；内部使用固定的 Vertex Key + GraphQL endpoint。
- 通过 `curl_cffi` 的 `impersonate` 轮换指纹；并实现了“会话老化阈值”机制：达到阈值后主动销毁会话并切换指纹。
- reCAPTCHA token 获取：访问 `/recaptcha/enterprise/anchor`，用 bs4 提取 token，再调用 `/reload` 获取最终 token。
- 失败处理：对 `Failed to verify action` 做了“复用同 token 再试一次”的专门分支；对 429 做退避；必要时会强制轮换 session。

### 5) OpenAIResponsesProvider（/v1/responses）

- 使用 `tools=[{type:'image_generation'}]` + `tool_choice` 强制走 image_generation。
- 能从两种结构取回 b64：
  - `object=='response'` 时读取 `output[].type=='image_generation_call'` 的 `result`
  - 兼容读取 `data[].type=='image_result'` 的 `b64_json`

### 6) OpenAICompatChatProvider（/v1/chat/completions 兼容）

- 请求体使用 `messages[].content` 的多模态格式：
  - 文本：`{type:'text', text:...}`
  - 图片：`{type:'image_url', image_url:{url:'data:image/png;base64,...', detail:'high'}}`
- 解析能力非常“宽松”以适配各种第三方：
  - Markdown 图片：`![](...)`
  - 纯 URL（正则提取）
  - data URI（image/video 的 base64）
  - 标准 JSON / SSE JSON chunk
- **视频**：如果解析到视频 URL，会返回 `{type:'video', url:'...'}`，上层会用 `Video.fromURL` 发送。
- **注意**：如果端点返回 base64 视频数据，插件不会直接发送，会提示“不支持直接发送内嵌视频数据”。

### 7) GenericImageProvider（siliconflow/bigmodel）

- 面向 `/images/generations` 一类端点，payload 为 `{model, prompt}`。
- 输入图最多携带 3 张：`image` / `image2` / `image3`（data:image/png;base64,...）。
- 返回：优先解析 `...url`。
  - 如果返回 `data:image/...;base64,...` 则直接解码
  - 如果返回远程 URL，则会二次下载

</details>

## 常见问题与排错

### 1) 提示“API 管线为空或无已启用的提供商”
- 你没有在 WebUI 的 `api_pipeline` 添加任何节点，或全部节点都被关闭（enabled=false）。

### 2) Vertex AI 匿名节点提示缺少 curl_cffi
- 该节点会返回："环境缺失 curl_cffi..."。
- 安装 `curl_cffi` 后再试（本仓库 requirements 已声明）。

### 3) 模板触发没反应
- 检查 `prefix=true` 时是否使用了前缀或 @机器人 唤醒。
- 检查模板名是否存在：`#画图帮助`。

### 4) 图生图提示“请发送或引用一张图片”
- 说明插件没有拿到任何底图：既没有图片、也没能拿到 @用户头像、也没能拿到发送者头像。
- 非 QQ 平台/ sender_id 非数字时，头像抓取会失败，这是预期行为。

### 5) 返回了视频但机器人没发图片
- 仅 `openai_compat_chat` 兼容端点可能返回视频 URL；插件会以“视频组件”发送。
- 如果端点返回的是 base64 视频数据，插件会提示当前不支持直接发送内嵌视频。

### 6) 插件如何保存持久化数据文件？
- `user_counts.json`：用户永久次数
- `group_counts.json`：群组永久次数
- `user_daily_counts.json`：用户每日额度使用情况（按日期）
- `group_daily_counts.json`：群组每日额度使用情况（按日期）
- `user_checkin.json`：签到日期记录（防止重复签到）
